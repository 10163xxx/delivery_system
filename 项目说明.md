# 外卖配送系统功能与架构说明

## 一、系统定位

本项目实现的是一个多角色外卖配送业务系统，核心目标不是只展示静态页面，而是把顾客下单、商家接单、骑手配送、售后评价、平台审核这些流程串成一个完整闭环。系统围绕四类角色展开：

- 顾客：浏览店铺、维护地址、下单、查看配送进度、评价与售后。
- 商家：入驻申请、店铺运营、菜单维护、订单处理、收入提现。
- 骑手：维护接单状态、承接配送任务、更新取餐和送达状态、收入提现。
- 管理员：处理商家审核、资格审核、评价申诉和售后工单。

系统的重点在于业务流程完整、前后端类型约束清晰，并引入地图定位、距离估算和配送路线展示能力，使项目更接近真实外卖平台的使用场景。

## 二、系统架构

项目采用前后端分离结构：

```text
React/TypeScript 前端
        |
        | HTTP JSON API
        |
Scala/http4s 后端
        |
        | JDBC / 持久化表
        |
PostgreSQL 数据库
```

前端负责页面展示、角色工作台、表单交互、地图展示和接口调用；后端负责 API 路由、会话校验、业务状态流转、配送报价、文件上传和数据持久化。前端登录后把会话 token 存在浏览器本地存储，请求后端时通过 `x-session-token` 传递，后端据此识别用户身份和角色权限。

### 前端模块

前端主要分为：

- `pages`：顾客端、商家端、骑手端、订单页、认证页等页面入口。
- `apis`：封装所有业务接口调用，例如创建订单、上传店铺图片、骑手提现等。
- `objects`：集中定义用户、店铺、菜单、订单、审核、地图坐标等类型对象。
- `features`：封装业务规则、校验、payload 构造和格式化逻辑。
- `system/api`：底层 HTTP 请求、API 路径定义、浏览器存储等基础能力。
- `public`：店铺分类图、测试图片、图标等静态资源。

前端调用链可以概括为：

```text
页面交互 -> 业务规则/对象类型 -> API 封装 -> HTTP 客户端 -> 后端接口
```

### 后端模块

后端主要分为：

- `routes`：总路由装配。
- `services/auth`：注册、登录、登出、会话读取。
- `services/customer`：顾客资料、地址、余额充值、状态查询。
- `services/merchant`：商家资料、店铺、菜单、图片上传、提现。
- `services/order`：订单创建、接单、拒单、出餐、分配骑手、取餐、送达、评价、售后和退款。
- `services/rider`：骑手资料、接单状态、提现。
- `services/admin`：商家审核、售后处理、资格审核和评价申诉处理。
- `system/app`：配送报价、地图默认数据等系统业务能力。
- `database/table`：数据库连接、表定义和数据读写。

后端调用链可以概括为：

```text
HTTP 路由 -> 会话/权限校验 -> 业务服务 -> 领域对象 -> 数据持久化
```

## 三、主要功能

### 1. 账号与多角色工作台

系统支持注册、登录、登出和会话恢复。账号带有角色信息，不同角色登录后进入不同工作台。顾客、商家、骑手和管理员看到的功能入口不同，避免把所有操作混在同一个页面里。

### 2. 顾客端功能

顾客端实现了较完整的点餐流程：

- 浏览店铺列表和店铺详情。
- 查看店铺菜单、菜品分类、规格选项和库存状态。
- 管理收货地址，并对地址进行地图定位。
- 选择地址后计算店铺到收货点的配送距离和配送费。
- 添加商品到购物车，选择规格，创建订单。
- 查看订单状态，包括待商家接单、备餐中、待骑手取餐、配送中、已完成等。
- 查看配送路线预览和预计送达时间。
- 对订单发起评价、售后申请或部分退款申请。

### 3. 商家端功能

商家端围绕店铺运营展开：

- 提交商家入驻申请。
- 设置商家资料、联系方式和收款账户。
- 管理店铺名称、地址、营业状态、营业时间等信息。
- 上传和展示店铺图片。
- 新增菜单项，维护菜品价格、分类、库存和规格选项。
- 处理顾客订单，包括接单、拒单、出餐。
- 查看店铺订单、评价和经营数据。
- 对顾客评价进行回复。
- 提交商家提现申请。

### 4. 骑手端功能

骑手端围绕配送履约展开：

- 维护骑手个人资料。
- 切换是否可接单。
- 查看可配送订单。
- 承接配送任务。
- 更新取餐状态和送达状态。
- 查看配送收入并提交提现申请。

### 5. 管理与审核功能

管理员端负责平台治理：

- 审核商家入驻申请。
- 处理资格审核。
- 处理评价申诉。
- 处理售后工单和退款请求。
- 维护平台订单流转的异常处理能力。

### 6. 订单、评价与售后闭环

订单不是简单生成记录，而是带有完整状态流转：

```text
顾客下单 -> 商家接单/拒单 -> 商家备餐 -> 骑手取餐 -> 骑手送达 -> 顾客评价/售后
```

系统还支持订单聊天消息、商家回复评价、售后处理和部分退款处理，使订单完成后仍能继续处理用户反馈。

## 四、地图与配送特色

地图能力是本项目的重点亮点之一。

### 1. 地址地理编码

顾客添加收货地址、商家维护店铺地址时，系统会尝试把文字地址转换成经纬度坐标。前端实现了地理编码缓存和多服务兜底：

- 优先使用高德地图地址解析，适配国内地址。
- 失败后使用 Photon、Nominatim 等公开服务作为备选。
- 对同一地址做本地缓存，减少重复请求。
- 设置超时时间，避免地图服务无响应时阻塞页面。

### 2. 坐标系统转换

由于国内地图服务常使用 GCJ-02 坐标，而部分开源地图瓦片使用 WGS-84 坐标，前端实现了 GCJ-02 到 WGS-84 的转换逻辑。这样可以减少坐标偏移问题，让标记点更准确地落在地图上。

### 3. 地图瓦片展示

前端封装了地址地图组件 `AddressTileMap`，用于展示顾客地址、商家地址和配送路线预览。地图组件支持：

- 主地址标记。
- 商家/顾客双点标记。
- 路线曲线展示。
- 加载状态和定位失败提示。
- 紧凑版与大图版展示。
- 雨天路况样式提示。

### 4. 配送距离与配送费

后端根据店铺坐标和收货地址坐标计算配送距离，并按距离区间生成配送费。距离计算使用球面距离公式，避免只用字符串地址做简单判断。系统还会根据配送距离判断是否可配送，距离过远时可以限制下单。

### 5. ETA 与路况提示

前端会结合订单状态、商家平均备餐时间和天气/时间窗口生成预计送达提示。例如雨天时会自动增加配送耗时，并在地图区域展示“雨天路况”提示，使配送信息更贴近真实业务。

## 五、类型与数据设计特点

项目把业务对象集中建模，而不是在页面或接口里随意写字符串和临时对象。典型对象包括：

- 用户与会话：用户 ID、用户名、密码、角色、session token。
- 商家与店铺：商家 ID、店铺 ID、店铺状态、店铺分类、营业时间、店铺坐标。
- 菜单：菜品 ID、分类、价格、库存、规格组和选项。
- 顾客：顾客资料、地址、默认地址、余额。
- 订单：订单 ID、订单状态、订单行项目、价格明细、聊天消息、售后和退款。
- 审核：商家申请、资格审核、评价申诉、处理结果。
- 地图：经度、纬度、配送坐标、店铺位置、顾客位置。

这种设计的好处是：前后端接口字段更明确，订单状态和角色权限更不容易写错，后续扩展新状态或新角色时也更容易定位修改点。

## 六、项目亮点总结

- 实现了顾客、商家、骑手、管理员四类角色，而不是单一页面演示。
- 订单链路完整，覆盖下单、接单、备餐、配送、评价、售后和退款。
- 引入地图定位、地址解析、坐标转换、配送距离计算、路线预览和 ETA 展示。
- 商家端支持真实运营功能，包括入驻、店铺资料、菜单、图片上传、订单处理和提现。
- 后端使用 PostgreSQL 持久化账号、店铺、菜单、订单和审核数据，服务重启后数据不会丢失。
- 前后端都使用强类型对象表达业务含义，减少魔法字符串和弱类型数据。
- 按业务域拆分代码目录，便于展示项目结构，也便于后续继续扩展。
