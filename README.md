# Project Setup

这个仓库包含两个独立部分：

- `frontend`: `Vite + React + TypeScript`
- `backend`: `Scala 3 + sbt + http4s`

当前配送业务只支持真实 Scala 后端，不再提供独立的 mock 后端脚本。

## 一键初始化

在仓库根目录执行：

```bash
./scripts/setup-dev.sh
```

这个脚本会：

- 调用 `frontend/scripts/setup-frontend.sh` 安装前端依赖
- 保留 `backend/scripts/setup-postgres.sh` 作为可选数据库示例脚本

如果你只想做其中一部分：

```bash
SETUP_DATABASE=0 ./scripts/setup-dev.sh
SETUP_FRONTEND=0 ./scripts/setup-dev.sh
```

## 启动项目

前端：

```bash
cd frontend
npm run dev
```

后端：

```bash
cd backend
./sbtw run
```

当前业务后端默认把认证状态和配送业务状态持久化到 PostgreSQL。

默认端口：

- 前端开发服务器：Vite 默认端口
- 后端：`http://127.0.0.1:8081`

## 代码约束

- 前后端都优先使用短小、单一职责的函数；函数过长时先拆辅助函数，再继续叠逻辑。
- 所有源码文件必须控制在 `300` 行以内；达到 `300` 行就视为需要继续拆分，不能再增长。
- 对象、函数、文件都不能无限增长；主文件只保留入口、装配和导出职责，超过一个屏幕仍看不清主流程时，必须继续拆到独立文件或独立对象。
- 禁止把几十个局部常量、分支和 JSX 区块堆回同一个页面文件；页面过长时优先拆 `...Content`、`...Section`、`...Parts` 或同职责子文件，避免一个文件同时承担路由入口、数据派生、表单交互和全部渲染细节。
- “拆分”指的是按职责拆成多个更小的子文件，不是把原文件整体平移到另一个同样巨大的文件里；如果 `...Content` 本身仍然很长，就必须继续拆。
- 文件命名必须表达职责，而不是简单在主文件名后追加 `Parts`、`Support`、`Content`、`Shared` 之类的模糊后缀；辅助文件名不能完整包含同目录主文件名，优先使用 `Sections`、`Fields`、`Groups`、`Copy`、`Helpers`、`Assembly` 等更具体的职责词。
- 对象也必须保持短小和聚焦；一个对象只表达一类配置、copy、枚举或字段集合，过长时继续按职责拆分，不能把整个页面的所有文案、字段、状态键和值域全塞进一个超大对象。
- 如果一个对象已经挂了十几个直接子项，默认说明职责过宽；应继续拆成更小的分组对象、子常量文件或按域命名的子对象，避免单个“消息桶”“配置桶”“文案桶”继续膨胀。
- 函数返回对象时同样不能做成大平铺桶；如果一个函数直接返回十几个字段、setter、action 或常量，必须先按职责拆成多个更小的返回分组，再由上层入口组合。
- TypeScript 源码里禁止使用 `interface`；统一使用集中收口的 `type`，并放到对应 `object` / `domain/object` 文件中。
- 前后端业务逻辑优先写成 pure function：相同输入得到相同输出，不直接读写外部状态；`IO`、网络、磁盘、数据库、时间、随机数、浏览器存储、定时器等副作用统一压到路由、存储、服务边界或显式 effect 包装层。
- 避免 OOP 式组织，尤其避免把行为塞进对象后再用 `obj.fn()` 分发业务流程；业务代码里不要设计“对象调用函数”的模式，优先把函数平铺成独立参数或普通函数调用。
- 优先使用显式数据传递和普通函数组合；状态装配可以集中，但不要把业务逻辑藏进“service”对象。
- 前后端代码默认使用强类型表达业务约束；禁止引入 `any`、`as any`、无约束裸对象载荷或用字符串硬编码代替领域类型，必要时优先补充类型定义再写逻辑。
- 前端与后端的类型定义必须集中放在对象目录里统一收口，禁止在业务流程文件、页面文件、动作文件或零散工具文件里临时定义公共类型；新增类型时优先放入对应模块的 `object` 或 `domain/object` 目录，再由业务代码引用。
- 后端同样必须保持强类型；禁止把业务参数、错误上下文或显示值回退到 `Any`、弱类型 JSON 中转或字符串拼接兜底，优先补充领域类型、枚举和类型类。
- 对外部输入、异常和反序列化结果，允许先用 `unknown` 接收，但必须在进入业务逻辑前完成显式收窄、校验或转换。
- 业务枚举值、角色名、状态名、路由段、存储键和默认行为不得在业务代码里裸写；统一收敛到领域常量、共享配置或类型安全的构造函数后再引用。
- 前端与后端的枚举类型禁止直接手写成裸字符串联合；必须先集中定义枚举常量或同等单一来源，再由类型系统从该来源推导类型，业务代码统一引用该枚举入口。
- 在强类型系统里禁止继续引入 magic number 和裸字符串；数值阈值、分页大小、默认金额、文案状态值、占位键名、时间窗口、校验边界和 UI 选择值都必须先抽到对象常量、领域常量或具名构造函数里，再由业务逻辑引用。
- 对象定义必须按职责集中收口：领域对象、请求/响应对象、枚举和共享类型统一放 `domain`；页面草稿、表单对象、工作台视图和界面编排对象统一放 `app`；组件目录只允许保留组件私有对象，不能继续散落公共业务对象。
- 页面、动作、服务、路由文件里禁止临时定义公共类型；发现页面里有本模块之外会复用的 `type`，必须先挪到 `object` 再继续写逻辑。
- 前端 `delivery` 目录只保留纯业务规则、校验、格式化、payload 构造和共享常量；带副作用的动作编排、页面流程装配和状态联动统一放 `app`；公共展示 copy 与视图元数据统一放 `components` 或具体 UI 目录，不再放回业务规则层。
- 前后端目录按业务模块保持镜像对齐；例如一侧新增 `auth`、`customer`、`merchant`、`order`、`review`、`admin`、`rider` 等模块时，另一侧也要在对应层级补齐同名目录或明确说明为何例外。
- 提交前检查前后端分层目录是否一致，至少对齐领域对象层、接口层（前端 `src/apis` / 后端 `src/main/scala/*/api`）以及业务装配层（前端 `src/pages`、`src/features` / 后端 `src/main/scala/*/app`）；前端不再单列 `services` 作为并行大类。
- 前后端不仅要模块名对齐，还要子职责目录对齐；如果前端已经拆出 `profile`、`menu`、`store`、`checkout`、`afterSales` 等子目录，后端对应域也应优先使用同名或同义职责子目录，而不是继续平铺在模块根下。
- 调整目录结构时，禁止只整理前端或只整理后端；涉及同一业务域的分组重构，默认同时检查并整理两侧目录层级，保持“域 -> 子职责 -> 文件”的对称结构。
- 同一父目录下禁止保留语义重叠或命名重叠的 sibling 目录；例如 `app` 与 `deliveryApp`、`app` 与 `appBuildRoleProps` 这类并存说明拆分没有收干净，必须继续并回统一主目录，再用子目录表达职责。
- 目录命名优先使用“主职责目录 + 子职责目录”的形式；应用编排统一归 `app/...`，不要再额外平铺一个带 `app` 后缀的新大类目录。
- 任一目录下禁止同时直接放源码文件和子目录；如果一个目录已经开始按子职责继续分层，就必须把该层剩余源码一并收进对应子目录，避免“半平铺半分组”的混合结构。
- 协议层与内部层必须明确区分：只有业务 request/response、前端直接消费的领域对象和业务 API 需要前后端 1:1 对齐；持久化对象、服务上下文、页面草稿和调试路由不得再混入协议层。具体规则见 `PROTOCOL_ALIGNMENT.md`。

## 开发流程

- 更新前端或后端后，如果需要重新看运行效果，先检查并结束旧开发进程，再启动新进程；不要复用旧进程观察新代码结果。
