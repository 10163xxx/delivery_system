# 外卖配送系统项目设计文件

## 一、业务角色设计

系统围绕四类角色设计：

```text
顾客 -> 下单与评价
商家 -> 接单与出餐
骑手 -> 取餐与送达
管理员 -> 审核与异常处理
```

四类角色共同组成外卖配送闭环。顾客产生订单，商家处理订单，骑手完成履约，管理员负责平台规则和异常处理。

## 二、核心业务流程

### 1. 顾客下单流程

```text
登录/注册
  -> 维护收货地址
  -> 地址地图定位
  -> 浏览店铺和菜单
  -> 选择菜品规格
  -> 计算配送距离和配送费
  -> 提交订单
  -> 查看订单进度和路线
  -> 评价或发起售后
```

### 2. 商家处理流程

```text
提交入驻申请
  -> 管理员审核通过
  -> 完善店铺资料和地图位置
  -> 维护菜单和库存
  -> 接收订单
  -> 接单或拒单
  -> 出餐
  -> 回复评价
  -> 申请提现
```

### 3. 骑手配送流程

```text
维护骑手资料
  -> 开启可接单
  -> 查看配送任务
  -> 接单
  -> 到店取餐
  -> 配送中
  -> 确认送达
  -> 结算收入
```

### 4. 管理员审核流程

```text
查看商家入驻申请
  -> 通过/驳回
  -> 处理资格审核
  -> 处理评价申诉
  -> 处理售后和退款工单
```

## 三、订单状态设计

订单状态按履约过程拆分：

```text
待商家接单
  -> 商家备餐中
  -> 待骑手取餐
  -> 配送中
  -> 已完成
```

异常分支包括：

- 商家拒单。
- 顾客发起售后。
- 顾客申请部分退款。
- 评价被申诉。
- 管理员介入处理。

这种状态设计让订单在不同角色端都有明确的可操作动作。例如待接单时商家可以接单或拒单，待取餐时骑手可以确认取餐，配送中时骑手可以确认送达。

## 四、地图功能设计

地图模块用于解决外卖系统里最关键的“位置”和“配送距离”问题。

### 1. 地址定位

顾客地址和商家地址都可以转换为经纬度坐标。系统先根据地址文本请求地图服务，成功后把坐标保存到对应对象中。没有定位成功的地址不能作为有效配送地址，店铺没有定位时也不能正常参与下单。

### 2. 地理编码兜底

地图解析按顺序尝试多个服务：

```text
高德地图 -> Photon -> Nominatim
```

这样设计是为了兼顾国内地址识别和公开服务可用性。如果一个服务失败，系统仍可尝试后续服务。

### 3. 坐标转换

系统考虑了 GCJ-02 与 WGS-84 坐标系统差异。国内地图服务返回的坐标可能与开源地图瓦片存在偏移，因此前端提供坐标转换逻辑，保证地图标记展示更准确。

### 4. 路线展示

地图组件支持展示：

- 顾客地址点。
- 商家店铺点。
- 两点之间的配送路线曲线。
- 配送阶段文案。
- 预计送达时间。
- 雨天路况提示。

地图不只是装饰，而是与下单、配送费、订单进度联动。

### 5. 配送费与可配送判断

后端根据店铺坐标和收货地址坐标计算距离，再按距离区间计算配送费。距离超过系统限制时，订单会被标记为不可配送，避免用户对过远地址下单。

## 六、接口设计

接口按业务域划分：

- `/api/auth/*`：注册、登录、登出、会话读取。
- `/api/delivery/state`：读取当前配送业务状态。
- `/api/delivery/customers/*`：顾客资料、地址、余额。
- `/api/delivery/merchants/*`：商家资料、店铺、菜单、图片、提现。
- `/api/delivery/riders/*`：骑手资料、接单状态、提现。
- `/api/delivery/orders/*`：订单创建、接单、出餐、配送、评价、售后。
- `/api/delivery/tickets/*`：售后工单处理。
- `/api/delivery/eligibility-reviews/*`：资格审核。

前端不在页面中直接拼接口地址，而是通过 `apis` 目录封装请求，再由 `system/api` 统一发送。

## 七、数据库与持久化设计

后端使用 PostgreSQL 持久化核心数据。需要长期保存的数据包括：

- 账号和登录会话。
- 顾客资料和地址。
- 商家申请、店铺、菜单。
- 骑手资料。
- 订单和订单状态。
- 售后、退款、评价和审核记录。

系统启动时会连接数据库，业务状态变化后写入数据库，因此服务重启不会丢失核心业务数据。

## 八、界面设计思路

界面按角色工作台组织，不把所有功能堆在一个页面中：

- 顾客端强调店铺浏览、购物车、地址地图、路线和订单进度。
- 商家端强调订单处理、菜单维护、店铺运营和收入。
- 骑手端强调可接单状态、配送任务和收入。
- 管理端强调审核列表、异常处理和操作反馈。

地图区域放在顾客地址、下单确认、订单进度等关键位置，用来辅助用户理解配送距离和配送状态。

## 九、设计特点

- 多角色设计完整，符合真实外卖系统的职责分工。
- 订单状态流转清晰，不同角色只看到自己可执行的动作。
- 地图功能参与真实业务，包括地址定位、距离计算、路线预览和可配送判断。
- 前后端对象按业务域组织，协议字段更清楚。
- 数据持久化覆盖核心业务对象，项目不是一次性的页面演示。
- 售后、退款、评价、申诉等后置流程补齐了订单完成后的业务闭环。
